# AI 问答编排规格

## Purpose
定义 C 端 AI 问答编排主链路，包括问题理解、任务规划、证据检索、证据排序、答案生成、质量校验、流式 workflow 事件、监控写入与结果归档。

## Requirements
### Requirement: 结构化 AI 问答编排链路

系统 MUST 将每次用户端问答请求处理为结构化编排流程，流程包括问题理解、任务规划、证据检索与工具执行、证据融合与排序、答案生成、质量校验和结果归档。

#### Scenario: 图谱增强问答成功

- **WHEN** 已认证用户提交图谱增强问答请求
- **THEN** 系统必须创建或确认会话，创建处理中消息，执行问答编排流程，持久化最终答案，并返回成功的聊天结果。

#### Scenario: 流式问答

- **WHEN** 已认证用户提交流式问答请求
- **THEN** 系统必须使用同一套编排流程输出 SSE 事件，且不能改变消息持久化和监控写入语义。

#### Scenario: 客户端可见 workflow

- **WHEN** 用户端问答请求被处理
- **THEN** 系统必须在聊天响应或 SSE 事件中暴露 workflow 状态、阶段状态、工具调用、耗时和归档标识。

#### Scenario: 会话记忆参与编排

- **WHEN** 用户端问答请求启用上下文模式
- **THEN** 编排流程必须在问题理解和答案生成 Prompt 构建时使用当前会话记忆上下文。

### Requirement: 问题理解

系统 MUST 调用已配置的大模型服务，并启用 JSON Mode，生成结构化问题理解结果。结果应包括问题重写、清洗后的上下文、标准化术语、扩展查询、意图、识别实体、复杂度和置信度。规则理解只能作为模型理解失败时的兜底方案。

#### Scenario: 油井工程术语标准化

- **WHEN** 用户问题包含油井工程别名或非标准术语
- **THEN** 系统必须在检索前标准化术语，并保留原始问题用于展示和审计。

#### Scenario: 无关上下文剔除

- **WHEN** 上下文模式启用且历史上下文包含无关轮次
- **THEN** 系统必须在检索和 Prompt 构建阶段排除无关上下文。

#### Scenario: 模型问题理解兜底

- **WHEN** 模型问题理解调用失败或返回无效结构化输出
- **THEN** 系统必须使用兜底理解结果，并将模型失败信息记录到编排归档。

#### Scenario: 问题理解 JSON Mode

- **WHEN** 系统调用模型执行问题理解
- **THEN** 请求必须要求返回 JSON 对象，系统必须在使用前校验解析后的结构。

### Requirement: 任务规划

系统 MUST 生成任务规划结果，用于判断是否需要图谱检索、后端托管内部工具、网络检索、多跳检索、问题分解以及子任务执行顺序。当前变更不得执行向量库检索；对外暴露时必须标记为禁用或未实现。

#### Scenario: 简单定义类问题

- **WHEN** 问题被识别为简单定义或解释类问题
- **THEN** planner 必须优先采用直接图谱检索，并跳过不必要的多跳查询或问题分解。

#### Scenario: 复杂机理对比类问题

- **WHEN** 问题需要对比、机理分析或涉及多个约束条件
- **THEN** planner 必须将问题标记为复杂，并为检索和答案生成生成有序子任务。

#### Scenario: 向量检索未实现

- **WHEN** planner 评估向量检索
- **THEN** 系统必须标记向量检索未实现，且不得调用任何向量数据库。

### Requirement: 证据检索与工具执行

系统 MUST 按照任务规划结果执行后端托管的内部工具，并将所有检索输出转换为标准化证据项。大模型服务不得直接访问 Neo4j、MySQL、向量数据库或任意网络资源。

#### Scenario: 需要图谱检索

- **WHEN** planner 判断需要图谱检索
- **THEN** 系统必须基于标准化问题、扩展查询和识别实体查询 Neo4j，并将图谱实体和关系返回为标准化证据。

#### Scenario: 网络检索关闭

- **WHEN** 网络检索被禁用
- **THEN** 系统不得执行网络访问，必须继续使用已启用的本地知识源。

#### Scenario: 工具调用被记录

- **WHEN** 系统调用内部检索工具
- **THEN** 系统必须记录工具名称、输入摘要、状态、耗时、输出摘要和失败原因。

#### Scenario: 后端工具白名单

- **WHEN** planner 请求工具调用
- **THEN** 系统必须只允许已注册的后端工具，并拒绝未知工具名称或非法参数。

### Requirement: 证据融合与排序

系统 MUST 在答案生成前对证据去重、识别冲突、重排序、应用来源权重并计算置信度。

#### Scenario: 重复证据

- **WHEN** 多个检索工具返回等价事实或重复图谱关系
- **THEN** ranker 必须保留一个标准证据项，并保留来源元数据用于追踪。

#### Scenario: 冲突证据

- **WHEN** 检索证据包含互相冲突的描述或结论
- **THEN** ranker 必须标记冲突，生成答案时必须说明不确定性或优先采用高置信来源。

### Requirement: 答案生成

系统 MUST 基于用户问题、任务规划结果、排序后的证据、相关上下文和系统 Prompt 生成答案。

#### Scenario: 存在可用证据

- **WHEN** 排序后的证据可用
- **THEN** 答案必须优先依据证据，而不是无依据地使用模型通用知识。

#### Scenario: 证据不可用

- **WHEN** 没有可靠证据可用
- **THEN** 答案必须明确说明知识库未提供充分依据，并给出保守回答或澄清问题。

### Requirement: 质量校验

系统 MUST 调用已配置的大模型服务，并启用 JSON Mode，校验生成答案是否回答了用户问题、是否在需要时引用证据、是否避免无依据断言、是否需要降级回答或澄清。规则校验只能作为模型校验失败时的兜底方案。

#### Scenario: 答案通过校验

- **WHEN** 生成答案覆盖用户问题且与排序证据一致
- **THEN** 系统必须将消息持久化为 SUCCESS。

#### Scenario: 答案需要降级

- **WHEN** 生成答案证据不足或存在较高幻觉风险
- **THEN** 系统必须返回降级答案，标记质量结论，并避免把无依据内容表述为事实。

#### Scenario: 需要澄清

- **WHEN** 问题过于模糊，无法安全回答
- **THEN** 系统必须返回追问澄清建议，而不是强行给出确定性答案。

#### Scenario: 模型质量校验兜底

- **WHEN** 模型质量校验调用失败或返回无效结构化输出
- **THEN** 系统必须使用兜底校验结果，并将校验失败信息记录到编排归档。

#### Scenario: 质量校验 JSON Mode

- **WHEN** 系统调用模型执行质量校验
- **THEN** 请求必须要求返回 JSON 对象，系统必须在应用降级或澄清决策前校验解析后的结构。

### Requirement: 答案流式兼容

系统 MUST 保持最终答案生成兼容 SSE 自然语言流式输出，不得强制流式答案 chunk 使用 JSON Mode。

#### Scenario: 流式答案生成

- **WHEN** 系统通过 SSE 生成答案
- **THEN** chunk 事件必须包含自然语言增量文本，最终 done 事件必须包含后端组装后的结构化 workflow 数据。

### Requirement: 结果归档与可观测性

系统 MUST 将最终和中间问答结果归档到专用编排归档表，并继续写入现有会话、消息、证据、监控和异常日志结构。

#### Scenario: 成功归档

- **WHEN** 问答请求成功完成
- **THEN** 系统必须持久化最终答案、证据摘要、workflow 阶段、工具调用、耗时、监控记录、归档记录和通过 requestNo 关联的依据来源数据。

#### Scenario: 失败归档

- **WHEN** 问答请求在任意编排阶段失败
- **THEN** 系统必须更新消息和请求状态，写入可用的部分 workflow 结果，更新归档记录，并在适用时创建异常日志。

### Requirement: 客户端 workflow 响应

系统 MUST 在保留现有响应字段的同时扩展 chat 和 SSE 响应中的 workflow 字段。

#### Scenario: 同步聊天响应

- **WHEN** 同步聊天完成
- **THEN** 响应必须包含 workflow 状态、当前阶段、阶段列表、工具调用列表、归档标识和可选的记忆快照。

#### Scenario: SSE 阶段事件

- **WHEN** 流式聊天中某个阶段开始、完成或失败
- **THEN** 系统必须发送 stage SSE 事件，事件包含 requestNo、sessionId、messageId、阶段编码、阶段名称、状态、耗时和失败原因。

#### Scenario: SSE 工具调用事件

- **WHEN** 流式聊天中某个内部工具调用开始、完成或失败
- **THEN** 系统必须发送 tool_call SSE 事件，事件包含工具名称、状态、耗时、输入摘要、输出摘要和失败原因。
