## Why

当前用户端 AI 问答链路仍以简单的图谱增强和大模型调用为主，缺少企业级问答系统需要的问题理解、任务规划、证据融合、质量校验和结果归档能力。为了支撑油井工程专业问答的准确性、可解释性和可观测性，需要将现有链路升级为可扩展的专业 Agent 编排流程。

## What Changes

- 新增 AI 问答编排流程，将一次问答拆分为问题理解、任务规划、知识检索与工具调用、证据融合排序、答案生成、质量校验、结果归档七个阶段。
- 问题理解阶段调用阿里百炼大模型完成问题重写、上下文剔除、术语标准化、查询扩展、意图识别、实体识别和复杂度识别。
- 新增 planner 决策能力，用于判断是否需要图谱、工具、多跳查询、问题分解，以及子任务执行顺序；向量库本次不实现，仅保留后续扩展边界。
- 扩展知识检索流程，本次以知识图谱查询为主，网络查询默认关闭，向量库查询不进入本次实现，并统一标准化检索证据。
- 新增 ranker 流程，对多源证据进行去重、冲突识别、重排序、来源权重计算和可信度评分。
- 强化答案生成与大模型质量校验，校验回答是否覆盖问题、是否引用证据、是否存在幻觉、是否需要降级回答或追问。
- 新增专门的问答编排归档表，保存完整阶段轨迹、工具调用、证据、质量结论、耗时和最终答案。
- 扩展 chat 与 SSE 响应结构，向前端返回本次 QA 的流程状态、阶段耗时、工具调用和归档标识，用于 UI 展示执行过程。

## Capabilities

### New Capabilities

- `ai-qa-orchestration`: 定义用户端 AI 问答的端到端专业 Agent 编排能力，包括问题理解、任务规划、检索、证据融合、生成、质检和归档。

### Modified Capabilities

无。

## Impact

- 影响用户端问答主链路：`/api/client/qa/chat` 与 SSE 流式问答接口，响应结构会新增流程状态和工具调用字段。
- 影响会话、消息、知识依据、监控埋点与异常日志写入逻辑。
- 可能新增或调整 AI 问答链路内部 DTO、service、prompt、planner、retriever、ranker、quality checker 等模块。
- 新增 MySQL 问答编排归档表及对应 mapper/entity/dto。
- 继续使用阿里百炼 OpenAI 兼容接口；问题理解、答案生成、质量校验可通过不同 prompt/参数调用模型完成。
