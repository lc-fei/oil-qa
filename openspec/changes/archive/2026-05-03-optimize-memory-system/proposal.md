## Why

当前 C 端问答的记忆机制只是读取当前会话全部消息后在 Java 中截取最近 3 条拼入 Prompt。该方案能支持简单追问，但无法在长会话中稳定保留早期信息，也缺少会话摘要、关键实体、当前主题和用户个性化偏好等结构化记忆 key。

## What Changes

- 新增 C 端会话记忆模块，统一负责维护当前会话内的滚动摘要、最近 2 轮原文窗口、摘要未完成溢出轮次和结构化记忆 key。
- 记忆只存在于当前会话，不跨会话，不形成用户长期画像。
- 每次发送消息时，将“会话摘要 + 摘要未完成的溢出问答 + 最近 2 轮原文问答 + 记忆 key”带入问题理解和 Prompt。
- 回答成功后维护滚动摘要：始终保留最近 2 轮原文，摘要覆盖更早的会话内容；第 3 轮起每轮成功后异步更新一次摘要。
- 记忆 key 包括关键实体、当前主题、用户在本会话内表达的个性化偏好、约束条件、待澄清问题等。
- 为本次 QA 编排归档记录保存使用的记忆快照，便于排查模型为何引用历史上下文。
- 不新增管理端持久记忆配置，不做跨会话用户长期记忆。

## Capabilities

### New Capabilities

- `conversation-memory`: C 端当前会话内的滚动摘要、最近 2 轮原文窗口、摘要未完成溢出轮次、记忆 key、Prompt 注入和归档能力。

### Modified Capabilities

- 无。

## Impact

- 后端代码：新增会话记忆 Service/DTO，调整 C 端 QA 主链路、问题理解入参、Prompt 构建、回答成功后的记忆维护和归档。
- Mapper：新增按会话查询最近 2 轮有效消息与摘要游标后的溢出消息 SQL，避免全量读取。
- 数据库：新增或扩展会话记忆存储，用于保存滚动摘要、summary 游标、recent window 信息和记忆 key。
- 接口：C 端请求参数保持兼容；workflow 可新增可选 memory 字段。
- 文档：更新用户端后端设计、接口文档、数据库设计和主对话流式分析文档。
